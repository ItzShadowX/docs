# Player Trails Operations

## Standard Admin Workflow

1. Confirm `/trails` registers and opens correctly.
2. Load a few categories to verify `trails.yml` parsed correctly.
3. Grant a trail with `/trails give` and revoke it with `/trails take`.
4. If using LuckPerms, confirm grants and revokes propagate immediately.
5. Reload once after config edits and confirm the economy and permission subsystems reconnect.

## Safe Live Changes

- Trail names, prices, and category edits: reloadable
- World allow or deny list changes: reloadable, but validate in the affected world immediately
- Permission provider changes: reloadable through `/trails reload`, but still test with a live account
- Asset-pack sensitive visual changes: restart recommended

## Troubleshooting

### Trails do not appear in a world

Check:

- `enabled-worlds`
- `disabled-worlds`
- whether the trail is enabled
- whether the player is actually running an active trail

### Players see trails but cannot use them

Check:

- the trail's `permission` node
- the active permission provider
- whether the player received a grant through `/trails give`
- whether the trail is disabled or assigned to the wrong category

### Purchasing is configured but staff cannot use a purchase command

That is expected in the current source snapshot. `PurchaseCommand` exists but is not registered in `TrailCommand`. Treat purchasing support as config-backed capability with dormant command support, not an active root command.

### Reload does not fix a bad state

`/trails reload` refreshes config, permission, and economy subsystems. If the issue persists after reload, inspect the provider plugin or the trail content itself rather than repeating reloads.
