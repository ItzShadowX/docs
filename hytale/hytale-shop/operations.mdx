# Hytale Shop Operations

## Standard Operator Workflow

1. Confirm `/shop` and `/sell` register at startup.
2. Open the default shop and one additional shop.
3. Complete one buy flow and one sell flow.
4. If stock is enabled, confirm stock changes after the trade.
5. If the editor is enabled for staff, open `/shop editor` and validate it loads cleanly.

## Safe Live Changes

- Price and item list edits: reloadable
- Additional shop definition edits: reloadable
- Stock tuning: reloadable, but validate affected items immediately
- Database backend swaps: maintenance window only
- Asset-pack or patchline-sensitive changes: restart and retest recommended

## Troubleshooting

### `/shop` works but an additional shop command fails

Check:

- the additional shop file exists in `shops/`
- the shop is enabled
- the generated command name does not collide with another root command
- permission checks in that shop config

### Buy or sell actions are inconsistent

Check:

- `enable-buying` and `enable-selling`
- inherited economy settings at global, shop, tab, and item levels
- item-level buy and sell permissions
- any configured buy or sell command hooks and sell-command conditions

### Stock values look wrong

Check:

- whether global stock is enabled in `stock.yml`
- item-level stock overrides in the active shop file
- whether `sell-replenishes` is expected to be on
- rotation settings that may be swapping visible inventory

### Reload changes behavior unexpectedly

Because Hytale Shop supports inheritance across multiple config layers, compare the global `config.yml` values against the specific shop or item file you changed. A reload issue is often an inheritance issue rather than a parser issue.
