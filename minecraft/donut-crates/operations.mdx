# Donut Crates Operations

## Standard Admin Workflow

1. Verify the plugin enabled and loaded all crate definitions.
2. Test `/crates open` on a staff account.
3. Inspect a player's keys before and after a grant.
4. Assign a real crate block and confirm the correct menu opens from interaction.
5. If holograms are enabled, verify placement and formatting in-world.

## Safe Live Changes

- Reward list edits: reloadable, but test the crate immediately afterward
- Hologram text changes: reloadable if your hologram backend is healthy
- Storage backend migration: maintenance window only
- Manual edits to assignment data: offline only

## Troubleshooting

### Crate opens but reward behavior is wrong

Check:

- the crate ID and filename still match
- reward materials and enchants use valid Paper names
- the reward slots in `crate-menu.items.reward.slots` fit the configured row count

### Keys are missing or not updating

Check:

- the active database backend and credentials
- that you are editing the correct `crateId`
- whether a previous migration left stale data in a different backend

### Assigned blocks stop working

Check:

- `databases/crates.yml` or the configured backend for the stored block mapping
- that the block still exists in the world
- whether the crate definition referenced by that assignment still exists

### Holograms do not appear

Check:

- whether `hologram.enabled` is true for the crate
- whether `DecentHolograms` or `FancyHolograms` is installed and enabled
- paragraph keys in `locale/en.yml`

## Upgrade Guidance

Treat crate definitions and assignment data separately. Crate files are content; assignment data is runtime state. Back up both before upgrades.
